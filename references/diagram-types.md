# 常用图型识别与绘制规范

文章地图完成后，先判断段落的信息结构。图型不是装饰风格，而是信息组织方式。

所有正文图型统一使用 16:9 横向画幅。构图规则必须在这个画幅内解决，不得因为节点较多或留白需求改用更高比例。

所有图型统一使用文章解释图语言：结构准确，但优先通过一个动作、连续路径或清晰空间关系来组织，不做模式分流。

## 一、图型路由

按以下顺序判断：

1. 是否存在明确时间或操作顺序？有则优先流程图。
2. 是否在解释一个系统由多个模块协作？有则优先架构图。
3. 是否围绕同一维度比较两个方案？有则优先对比图。
4. 是否强调多个对象之间的影响、依赖或映射？有则优先关系图。
5. 是否在拆解一个整体的内部组成或层级？有则优先结构图。
6. 以上都不是，或重点是情绪、隐喻和转折时，使用概念插图。

## 二、流程图

### 识别信号

先提出假设，再筛选，然后找证据，最后生成结果。

### 必备字段

1. 起点。
2. 中间状态。
3. 方向。
4. 终点。
5. 循环或回退条件，如有。

### 构图规则

1. 默认从左到右。
2. 三至五个主节点最易读，最多七个。
3. 一个箭头只表达一种方向。
4. 循环只画一条回路。
5. 节点形状保持一致，关键节点用橙色强调。

### 橙猫用法

橙猫只能位于关键动作节点，例如筛选、检查、确认。不要每个节点都出现。

## 三、架构图

### 识别信号

输入经过不同模块或层，形成输出。文章出现系统、模块、服务、模型、数据层、接口等词。

### 必备字段

1. 系统边界。
2. 输入和输出。
3. 核心模块。
4. 模块之间的数据流。
5. 层级或归属。

### 构图规则

1. 优先左输入、中央处理、右输出。
2. 分层架构使用上下层级，不与左右流程混用。
3. 每层最多四个模块。
4. 实线表示主数据流，虚线只表示辅助关系。
5. 不画代码细节和装饰性设备。

### 橙猫用法

橙猫可站在系统入口、核心模块或观察台，提供尺度和品牌识别，不参与所有数据流。

## 四、对比图

### 识别信号

传统与新方法、相关性与推理、过去与现在、优点与限制。

### 必备字段

1. 共同输入或共同问题。
2. 左右两个方案。
3. 同一比较维度。
4. 不同结果。

### 构图规则

1. 左右严格对齐。
2. 两侧使用相同数量和位置的视觉单位。
3. 只比较一至三个维度。
4. 中央可放共同输入，避免两边重复。
5. 橙色只强调关键差异，不给某一侧全部染色。

### 橙猫用法

橙猫可以站在中央比较点，或在其中一侧执行关键动作，但不能遮挡对比项。

## 五、关系图

### 识别信号

对象之间存在影响、依赖、因果、相似、映射、传播或网络关系。

### 必备字段

1. 核心对象。
2. 次级对象。
3. 关系方向。
4. 关系类型或强弱。

### 构图规则

1. 只保留一个中心，或明确没有中心的网络。
2. 主对象不超过七个。
3. 线条交叉超过两处时，重新布局。
4. 方向有意义时必须使用箭头或明显流向。
5. 线条粗细只在原文确实表达强弱时使用。

### 橙猫用法

橙猫可连接两个关键对象，或观察中心对象。不要让猫替代关系线。

## 六、结构图

### 识别信号

文章在解释一个概念、产品或系统内部由哪些部分组成，以及这些部分如何包含或分层。

### 必备字段

1. 整体边界。
2. 一级组成。
3. 必要的二级组成。
4. 包含或层级关系。

### 构图规则

1. 使用剖面、容器、分层或嵌套。
2. 一张图只展示一个拆解层级。
3. 一级组成最多五个。
4. 不同时使用树形和剖面两种结构。
5. 组成部分使用清晰边界，避免漂浮图标。

### 橙猫用法

橙猫可掀开外壳、指向内部或站在剖面旁提供尺度。

## 七、概念插图

用于情绪、隐喻、案例和转折。遵循 `conception-protocol.md` 的主动作、关系语法和读图测试。

## 八、混合图限制

允许以下轻量组合：

1. 流程图加局部对比。
2. 架构图加一条主数据流。
3. 结构图加内部关系。
4. 概念插图加一个简单流程动作。

不允许：

1. 流程、架构、关系、对比全部混在一张图。
2. 用大量图标替代逻辑。
3. 用橙猫逐步表演完整系统。
4. 依赖长段文字标签才能读懂。

## 九、重复元素预算

结构型内容需要视觉一致性，但不能靠大量复制实现。

1. 大型同形模块最多 3 个，超过时改为一层容器或一个集合。
2. 同形内容卡最多展示 3 张，其余用重叠轮廓、扇形堆叠或一个带数量感的集合表示。
3. 数据点最多突出 5 个，其余用一条轨迹、密度带或局部点群表示。
4. 流程超过 5 步时，优先分组为 3 个阶段，或拆成两张图。
5. 对比图的两侧保持结构对齐，但内部图形应服务各自含义，不复制同一套图标。
6. 架构图中的同类实例用一个模块加叠层边缘表示，不横排多个相同盒子。
7. 关系图中的重复叶子节点可合并为一个分组，不画放射状同形花盘。
8. 结构图中的同类组成用分区表达，不用漂浮的重复图标填满空间。

允许重复的情况：

1. 重复本身就是文章重点，例如候选从多变少。
2. 数量差异是核心比较维度。
3. 阵列用于表达规模或洪流。

即使允许重复，也只展示足够读懂差异的最少数量。

## 十、视觉重量

1. 单个普通模块面积不超过画面的 15%。
2. 核心模块不超过约 20%。
3. 对比图两侧主要结构的外接面积尽量接近，差异不超过约 1.5 倍。
4. 复杂结构使用较少细节，简单结构可以略加轮廓变化，避免一侧过重。
5. 大模块不同时使用粗边框、密集内部元素和橙色填充。
6. 箭头和连接线不能比模块轮廓更抢眼。
7. 橙猫面积仍保持约 6% 至 10%。

## 十一、手写中文标签

文章解释图可以使用少量手写中文标签：

1. 每张图 2 至 5 个。
2. 每个标签优先 2 至 4 字，最多 6 字。
3. 标签用于命名节点，不写解释句。
4. 标签位置靠近对象，避免悬浮在大片空白中央。
5. 字体气质自然、清晰、略有手写感，不用儿童涂鸦体或正式印刷标题。
6. 同层标签大小一致，关键标签只允许轻微放大。
7. 必须逐字校对，错误文字视为不合格。

完整标签规划和颜色语义见 `annotation-system.md`。

## 十二、避免 PPT 感

1. 不在左上角放流程图、系统架构等类型标题。
2. 不给所有模块加相同圆角盒和相同阴影。
3. 不使用规则网格填满画面。
4. 不使用大量等粗箭头。
5. 允许模块轮廓保持一致，但内部结构和留白节奏必须服务含义。
6. 流程可以用连续路径、物理动作或阶段容器表达，不必总是一排卡片。
7. 核心结构整体应占画面约 40% 至 60%，其余空间用于留白和阅读方向。
8. 单个模块仍遵循不超过 15% 至 20% 的视觉重量限制。

## 十三、结构表达 QA

1. 图型是否匹配段落信息结构？
2. 一眼能否看出阅读方向？
3. 主节点是否不超过七个？
4. 连接线是否有明确含义？
5. 同类节点是否使用一致形状？
6. 是否删除了不承担信息的装饰？
7. 缩小到手机宽度后是否仍能辨认？
8. 橙猫是否参与关键含义，而不是遮挡信息？
9. 相似元素是否超过理解所需的最少数量？
10. 画面是否先呈现核心关系，而不是先呈现规则阵列？
11. 单个元素是否因体量、线条、细节或颜色过强而失衡？
12. 对比结构的视觉重量是否大致对等？
13. 手写标签是否准确、克制且真正帮助理解？
14. 核心结构整体是否占画面约 40% 至 60%，既不空散也不拥挤？
15. 是否出现规则网格、同形圆角盒阵列或大量等粗箭头造成的 PPT 感？
